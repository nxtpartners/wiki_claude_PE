Almost every disappointing answer from Claude traces back to a handful of avoidable habits, and they are all about context. None of them is hard to fix once you can name it. This page is the short list of what goes wrong most often, why it hurts, and what to do instead.

## The recurring mistakes

<ComparisonTable title="What goes wrong, and the fix" caption="Six habits that weaken output, each with a one-line correction.">

| Mistake | Why it hurts | Do this instead |
| --- | --- | --- |
| **Vague asks** ("what do you think?") | Claude has no task shape, so it hedges and generalises | State a role, a task, and a format: "assess these three things, in this order, as bullets" |
| **Dumping a file with no instructions** | Claude guesses what you want and usually guesses wrong | Attach the file and say exactly what to do with it |
| **Trusting unverified output** | Fluent prose can be confidently wrong; errors reach the IC | Check every figure and claim against the source before use |
| **Overstuffing one chat** | Unrelated tasks pollute the context and confuse the answer | One chat per job; start a new chat for a new task |
| **Not iterating** | The first answer is rarely the best one available | Push back: refine, expand, challenge, in the same chat |
| **Forgetting to specify format** | You get a shape you then have to rewrite | Say "a table", "four headings", "five bullets" up front |

</ComparisonTable>

## Dumping a file with no instructions

This is the most common one, so it is worth seeing directly. Attaching a sixty-page CIM with the word "thoughts?" tells Claude almost nothing. It will produce a generic overview that may miss the thing you actually cared about.

<PromptCard title="The fix: attach, then instruct">
You are assisting a KPMG private equity associate. I have attached the Project Atlas teaser.

Give me a one-page initial screen under four headings (business, financials, investment angle, key risks), as short bullets. Cite the page for any figure, and write "not stated" rather than estimating anything the teaser leaves out.
</PromptCard>

The file is the same. The instruction is what turns a shrug into a usable screen.

## One chat per job

Context accumulates inside a conversation, which is a strength when every message builds on the last and a liability when you have changed the subject three times. A chat that drifts from a CIM summary, to an unrelated market question, to a draft email, gives Claude a muddled backdrop for the fourth request. Keep one chat to one task; start a fresh one when the job changes. It costs nothing and keeps every answer on clean ground.

## The mistake that actually matters

Most items above cost you a weak draft and a few minutes. One of them costs you credibility: trusting output you did not verify. Claude's writing always reads fluently and authoritatively, even when a figure is invented or a name is slightly wrong. That polish is precisely what makes an unchecked number dangerous, because nothing on the surface signals the error.

<Callout type="warning" title="Verify before anything leaves Claude">
Never lift a figure, name, or claim from Claude into a deliverable without checking it against the original source. This is not optional and it does not get easier to skip as you get faster; it gets more tempting and more risky. Verification is the one habit that protects every other use of the tool. The discipline is set out in full in [Verify Everything](/best-practices/verify-everything).
</Callout>

## The takeaway

The fixes are small and they compound: brief Claude properly, keep one chat to one job, specify the format, iterate past the first answer, and verify everything before it leaves the screen. Avoid these six habits and most of the frustration with the tool simply disappears.
